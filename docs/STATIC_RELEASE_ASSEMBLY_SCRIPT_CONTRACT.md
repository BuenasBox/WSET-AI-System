# Static Release Assembly Script Contract

Phase: 4A.3.7.20.6

Status: contract only. This phase does not create the assembly script, does not
create `dist/`, does not copy files, does not modify frontend sources, and does
not deploy.

## 1. Purpose

A future release assembly script will assemble a deployable static site
containing:

- the architecture dashboard root;
- the Diagnostic SBA Lab under `/diagnostic-sba/`.

The script must assemble only explicitly allowed static frontend assets. It
must not copy unsafe repository materials such as knowledge corpus files,
private learner state, drafts/reviews, docs, tests, tools, or runtime artifacts.

The assembled release remains:

- private lab;
- static only;
- training-only;
- noindex/nofollow;
- no official WSET evaluation;
- no examiner scoring;
- no learner persistence.

## 2. Future Script Path

Recommended future script path:

```text
tools/dashboard/assemble_static_release.py
```

This phase does not create that file.

## 3. Future Output Directory

Recommended future output directory:

```text
dist/epistemiclab-static/
```

This directory should be generated by the future script and ignored by git
unless a later phase explicitly decides otherwise.

The future script must fail closed if asked to write outside `dist/`.

## 4. Input Directories

Allowed input directories:

```text
frontend/architecture-dashboard/
frontend/diagnostic-sba/
```

The future script must not read deployment assets from `knowledge/`, `docs/`,
`tests/`, `tools/`, source-bank locations, private memory folders, or runtime
artifact folders.

## 5. Required Output Tree

The assembled output tree must be:

```text
dist/epistemiclab-static/
  index.html
  system_state.json
  robots.txt
  CNAME
  .nojekyll
  diagnostic-sba/
    index.html
    preguntas.json
```

The future script must validate that this exact required tree exists after
assembly.

## 6. Allowlist Policy

The future script must copy only explicitly allowed files.

Allowed dashboard files from `frontend/architecture-dashboard/`:

```text
index.html
system_state.json
robots.txt
CNAME
.nojekyll
```

Allowed Diagnostic SBA files from `frontend/diagnostic-sba/`:

```text
index.html
preguntas.json
```

No directory-wide recursive copy is allowed unless every copied file is checked
against the allowlist first.

## 7. Denylist Policy

The future script must never copy:

```text
knowledge/
docs/
tests/
tools/
.git/
.codex/
.claude/
node_modules/
__pycache__/
.pytest_cache/
*.pyc
```

It must also never copy:

- private learner state;
- `knowledge/nazareth/`;
- diagnostic SBA drafts/reviews;
- source question bank files;
- research docs;
- runtime artifacts;
- self-eval reports or attempts;
- retrieval-sandbox outputs;
- exporter scripts;
- schemas;
- local worktree metadata;
- deployment caches such as `.vercel/`.

The denylist is a backstop. The allowlist is authoritative.

## 8. Safety Checks

The future script must fail closed if:

- any required file is missing;
- the output path is outside `dist/`;
- any attempted copy is outside the allowlist;
- `robots.txt` is missing;
- dashboard `noindex` / `nofollow` metadata is missing;
- Diagnostic SBA Lab `noindex` / `nofollow` metadata is missing;
- `diagnostic-sba/preguntas.json` is missing;
- `preguntas.json` is not valid JSON;
- `preguntas.json` fails static demo payload validation;
- `preguntas.json` contains pre-submit leakage in `items[]`;
- the release tree contains forbidden files;
- the assembled lab route would lack `diagnostic-sba/index.html`;
- the assembled lab data route would lack `diagnostic-sba/preguntas.json`.

Fail closed means the script must stop, report the violation, and write no
partial release when possible.

## 9. Modes

The future script should support:

```text
--dry-run
--write
--clean
```

Default mode must be dry-run/no write.

Mode expectations:

- `--dry-run`: inspect inputs, calculate planned copies, validate expected
  output shape, print a summary, and write nothing.
- `--write`: assemble the release tree only after all safety checks pass.
- `--clean`: remove only the generated `dist/epistemiclab-static/` directory,
  after verifying the resolved path is inside `dist/`.

## 10. Validation Requirements

The future script should validate:

- release tree shape;
- no forbidden files;
- `preguntas.json` validates with the static demo export payload validator;
- dashboard `noindex,nofollow` is present;
- lab `noindex,nofollow` is present;
- `robots.txt` contains a disallow-all rule;
- no sitemap is present;
- no analytics or tracking references exist;
- no backend/API/Supabase references exist;
- no external dependencies if feasible by static inspection.

The future implementation may reuse existing static demo export validation
helpers where appropriate, but it must not mutate the source payload.

## 11. Deployment Boundary

The future script must not deploy.

It only assembles files into:

```text
dist/epistemiclab-static/
```

Deployment remains a separate manual/explicit step. The script must not run:

```text
git subtree push
git subtree split
git push origin ... gh-pages
```

The script must not modify `gh-pages`, `CNAME`, dashboard source files,
Diagnostic SBA source files, `preguntas.json`, or any backend/runtime files.

## 12. Governance Requirements

The assembled release must remain:

- private lab;
- noindex/nofollow;
- no official WSET evaluation;
- no official WSET score;
- no pass/fail;
- no examiner scoring;
- no examiner authority;
- no certification-readiness claim;
- no learner persistence;
- static only;
- no backend;
- no API;
- no Supabase;
- no analytics or tracking.
